# Supabase migration 同步（Roamie）

遠端與本地 `supabase/migrations` 常因 Lovable / Dashboard / 多分支而**紀錄不一致**。  
**不要** `db reset`、不要刪 `profiles`。

---

## 錯誤 A：`duplicate key schema_migrations_pkey`（20260522160000 already exists）

遠端已記錄該 version，CLI 又試圖 INSERT。

```bash
supabase migration repair --status applied 20260522160000
```

---

## 錯誤 B：`Found local migration files to be inserted before the last migration on remote`

代表：遠端**最後一筆** migration 的 version **比**本地尚未登記的檔案還新；中間這些檔案無法用一般 `db push` 插入：

| Version | 檔案 |
|---------|------|
| 20260522160100 | profile_plan_tier |
| 20260523120000 | saved_collections |
| 20260524120000 | fix_profile_media_rls |
| 20260525120000 | ensure_saved_places |
| 20260525130000 | ensure_saved_trips |
| 20260527171500 | fix_profiles_updated_at |
| 20260527180000 | trip_place_image_metadata |
| 20260527200000 | trip_custom_title_cover |
| 20260527210000 | profile_travel_survey_columns |

### 做法 1（建議）：只補測驗欄位 + 修復紀錄

**不依賴** `db push` 跑中間 migration。

1. SQL Editor 執行：
   - `verify_profile_survey_columns.sql`
   - `apply_profile_travel_survey_columns.sql`
   - 再 verify 一次

2. 若遠端 schema **本來就有** 上表其他 migration 的變更（多數 Lovable 專案已具備），一次標記 gap：

```bash
chmod +x supabase/scripts/repair_migration_gap.sh
./supabase/scripts/repair_migration_gap.sh
```

或手動：

```bash
supabase migration repair --status applied 20260522160100
supabase migration repair --status applied 20260523120000
supabase migration repair --status applied 20260524120000
supabase migration repair --status applied 20260525120000
supabase migration repair --status applied 20260525130000
supabase migration repair --status applied 20260527171500
supabase migration repair --status applied 20260527180000
supabase migration repair --status applied 20260527200000
supabase migration repair --status applied 20260527210000
```

3. 確認：

```bash
supabase migration list
```

Local / Remote 應對齊，之後 `db push` 不應再報「inserted before」。

### 做法 2：遠端真的缺少中間 migration 的 schema

在確認**需要**補跑 SQL 時，才用（有風險，請先備份）：

```bash
supabase db push --include-all
```

會依時間順序套用所有尚未登記的 migration。  
若某支失敗，改在 Dashboard 手動執行該支 `.sql` 後，對該 version 做 `migration repair --status applied`。

## 測驗欄位 idempotent SQL

- 檢查：`supabase/scripts/verify_profile_survey_columns.sql`
- 套用：`supabase/scripts/apply_profile_travel_survey_columns.sql`

`travel_personality` 若已存在不會改動；其餘為 `ADD COLUMN IF NOT EXISTS`。
